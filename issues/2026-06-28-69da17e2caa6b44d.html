GPT-5.6 Sol 프리뷰와 미국 정부 요청에 따른 '신뢰 파트너 한정' 배포

blog_post · 원문 · 3 sources

OpenAI가 GPT-5.6 Sol·Terra·Luna 세 모델의 프리뷰를 공개했어요(https://openai.com/index/previewing-gpt-5-6-sol/). Sol이 플래그십, Terra가 일반용, Luna가 저비용·저지연 모델이에요. 가격은 입력·출력 100만 토큰당 Sol 5달러/30달러, Terra가 그 절반, Luna 1달러/6달러로 책정됐어요. Sol에는 두 개의 새 모드가 들어가요. 하나는 "max" 추론 모드로, 같은 모델이 더 긴 사고 트레이스를 돌려요. 다른 하나는 "ultra" 모드인데, Sol 한 모델 안에서 협력 서브에이전트를 같이 굴려 하드한 멀티스텝 태스크를 푸는 구조예요. OpenAI는 코딩·바이오·사이버 분야에서 에이전트 능력이 가장 크게 올랐다고 적었어요.

뉴스의 중심은 모델 자체가 아니라 배포 조건이에요. CNBC와 TechCrunch는 OpenAI가 GPT-5.6 접근을 "신뢰 파트너 소그룹"에만 열었고, "참여자 명단을 미국 정부와 공유한" 상태라고 보도했어요(https://www.cnbc.com/2026/06/26/openai-limits-new-ai-models-to-trusted-partners-request-us-government.html). 케이스별 접근을 요청한 쪽은 미국 상무부예요. OpenAI는 같은 시점에 "이런 종류의 제한이 일반 표준이 되어서는 안 된다"는 입장을 함께 냈어요. ChatGPT·Codex·API의 일반 사용자에게는 추후 단계적으로 풀린다고 적었어요.

선행 흐름과 비교하면 두 단계의 변화가 보여요. 2주 전 다룬 GPT-5.5는 출시 직후 일반 API에 풀린 모델이었어요. 한편 3주 전 다룬 Claude Fable 5는 미국 정부가 외국인 접근을 차단한 사례였어요. GPT-5.6은 한 발 더 들어가서, 출시 단계에서부터 "정부가 명단을 본 파트너만" 들어갈 수 있는 게이트를 깔았어요. 모델 자체로는 GPT-5.5 → Sol로 가는 한 세대 폭의 업데이트로 읽혀요. 다만 ultra 모드의 서브에이전트 협력은 같은 추론 호출 안에 멀티에이전트 토폴로지를 박는 첫 메이저 시도라, 자체 멀티에이전트 프레임워크를 굴리는 팀이라면 같은 워크로드를 ultra 모드 한 번으로 처리하는 게 더 싸고 빠른지 따져볼 만해요.

엔지니어 관점에서 짚어둘 점은 두 가지예요. 하나는 모델 선택이 이제 "어떤 능력을 살까"뿐 아니라 "참여 자격이 되는가"까지 포함된다는 점이에요. 다른 하나는 ultra 모드처럼 모델 안에 들어간 멀티에이전트는 외부 프레임워크 기반 멀티에이전트와 비용·관찰 가능성·디버깅 면에서 트레이드오프가 다르다는 점이에요. 자기 팀이 신뢰 파트너 명단에 들어 있다면, 운영 중인 SWE-bench Verified·Terminal-Bench 트레이스를 Sol의 ultra 모드로 한 번 그대로 흘려보고 같은 태스크에서 GPT-5.5 단일 호출 대비 토큰 비용 대비 통과율이 실제로 이득인지부터 측정해보세요.

Sakana Fugu: 프런티어 모델 여러 개를 한 엔드포인트 뒤로 묶는 7B 컨덕터

blog_post beta · 원문 · 2 sources

Sakana AI가 6월 22일 Fugu와 Fugu Ultra를 정식 출시했어요(https://sakana.ai/fugu/). 4월 25일부터 베타로 돌리던 모델을 GA로 풀었어요. 핵심은 한 번의 요청을 받아 어떤 프런티어 모델 묶음을 활성화할지 결정하는 7B "컨덕터" 모델이에요. 활성화 풀에는 GPT-5.5, Claude Opus, Gemini 3.1 Pro가 들어가요. 컨덕터는 단순 분류 기반 라우터가 아니라, 모델 간 중간 결과를 어떻게 주고받을지·그 결과를 어떻게 합칠지까지 학습된 정책으로 결정해요. 엔드포인트는 OpenAI 호환이라, 기존 SDK를 그대로 두고 base URL만 갈아끼우면 돼요. 컨덕터는 ICLR 2026에 실린 두 논문 TRINITY와 the Conductor를 토대로 학습됐다고 적었어요.

두 가지 티어가 있어요. Fugu는 저지연 일상용으로 튜닝된 모델이에요. Fugu Ultra는 더 큰 에이전트 풀을 가동해 하드한 멀티스텝 태스크에서 품질을 우선하도록 튜닝됐어요. 가격은 Fugu Ultra가 입력·출력 100만 토큰당 5달러·30달러예요. 구독 플랜은 월 20달러·100달러·200달러 세 구간으로 나뉘어요. 같은 주에 발표된 GPT-5.6 Sol의 ultra 모드가 한 벤더 안에서 서브에이전트를 묶는 그림이라면, Fugu는 여러 벤더의 프런티어 모델을 같은 토폴로지로 묶는 그림이라는 점이 다르거든요.

가장 가까운 비교군은 Martian이나 NotDiamond, OpenRouter의 라우터 계열이에요. 이쪽은 보통 프롬프트 분류기로 한 요청을 한 모델로 보내요. Fugu의 컨덕터는 한 요청을 여러 모델로 쪼개고 중간 결과를 받아 합치는 정책까지 같이 학습돼 있어요. 라우터가 "어디로 보낼까"만 푼다면 컨덕터는 "어디로 보내고, 어떻게 합칠까"까지 들고 가요. ICLR 2026 TRINITY 논문이 같은 태스크를 여러 모델이 동시에 푸는 동기적 합의 패턴을, 같이 게재된 the Conductor 논문이 학습된 합의 정책을 다뤘다고 적혀 있어요. 정리하면 Fugu는 학습된 멀티모델 합의 정책을 한 7B 모델 안에 응축해 둔 형태예요.

채택 신호는 두 갈래로 봐야 해요. 하나는 GA 첫 한 주의 외부 사례 폭이고, 다른 하나는 OpenAI 호환 인터페이스가 실사용 워크플로에 얼마나 매끄럽게 끼는가예요. 자기 스택에 적용해본다면, 운영 중인 에이전트 한 개의 비프로덕션 트래픽 슬라이스를 Fugu Ultra 엔드포인트로 그대로 흘려보고, 같은 평가셋에서 토큰 비용 대비 정답률이 기존 단일 모델 베이스라인보다 실제로 움직이는지부터 측정해보세요.

Claude Tag: 채널 단위 공유 메모리와 'ambient' 모드를 가진 Slack 전용 에이전트

blog_post beta · 원문 · 2 sources

Anthropic이 6월 23일 Claude Tag을 베타로 공개했어요(https://www.anthropic.com/news/introducing-claude-tag). Claude Enterprise·Team 플랜 고객 대상이고, 안에서 도는 모델은 Claude Opus 4.8이에요. 기존 "Claude in Slack" 앱과 가장 큰 차이는 단위에 있어요. 옛 앱은 사용자별 비휘발성 없는 1:1 세션이었어요. Claude Tag은 채널 단위로 단 하나의 Claude가 떠 있어요. 같은 채널의 모든 사람이 같은 인스턴스를 부르고, Claude는 그 채널의 작업 맥락을 시간 축으로 누적해 들고 가요. 관리자가 사전에 정의한 도구 묶음·MCP 서버와도 채널마다 다른 범위로 연결돼요.

새로 들어간 동작은 두 가지예요. 하나는 비동기 작업이에요. 사용자가 태그한 작업을 Claude가 단계로 쪼개 시간이 걸리는 일은 시간·일 단위로 굴려요. 결과는 같은 스레드에 다시 게시해요. 다른 하나는 "ambient" 모드예요. 옵션으로 켜면 Claude가 정체된 스레드를 다시 핑하거나, 새 주제가 올라오면 관련 맥락을 끌어와 붙이거나, 장시간 작업의 중간 상태를 주기적으로 보고해요. 다만 명시적으로 권한이 주어지지 않은 비공개 채널은 ambient 모드라도 읽을 수 없다고 적혀 있어요. The Decoder는 Anthropic이 "내부 프로덕트 팀이 작성하는 코드의 약 65%가 사내 버전 Claude Tag을 거쳐 생성된다"라고 인용했어요(https://the-decoder.com/claude-tag-embeds-anthropics-ai-in-slack-already-writes-65-percent-of-internal-code-company-says/).

선행 흐름과 견주면, 지난주 다룬 Block의 BuilderBot이 자체 인프라 위에 Goose를 얹어 Slack에서 코딩 에이전트를 굴리는 사례였어요. 차이는 두 군데예요. Block은 사용자가 호출하는 도구 호출 묶음이었어요. Claude Tag은 채널이 호출하는 공유 인스턴스예요. 가까운 다른 비교군은 GitHub의 Padawan이나 Cursor의 백그라운드 에이전트인데, 둘은 IDE 또는 PR 기반이에요. Slack 채널을 1차 표면으로 끌어다 쓴다는 점이 Claude Tag의 차별점이에요. 한편 기존 Claude in Slack 앱은 8월 3일에 회수돼요. 관리자는 그 사이 30일 안에 채널 접근, 도구 연결, 메모리 범위를 새로 잡아야 해요.

리스크 관점도 짧게 짚어둬요. 채널마다 별 토큰 예산 상한을 걸 수 있고, 도구·MCP는 채널 범위로 분리할 수 있어요. 두 손잡이를 무시하면 같은 봇이 동시에 데이터 누수와 토큰 소진 모두 일으킬 수 있어요. 자기 조직에 도입한다면, 우선 한 저위험 채널에서 도구·MCP를 일체 연결하지 않은 채 ambient 모드만 24시간 켜본 다음, Anthropic 감사 로그에서 Claude가 재핑·맥락 호출을 어디서 했는지부터 확인해보세요. 도구 하나 붙이기 전에 소음 수준을 먼저 가늠하는 셈이에요.
