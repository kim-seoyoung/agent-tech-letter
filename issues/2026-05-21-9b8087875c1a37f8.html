한 주에 세 곳에서 동시에 뜬 Agent Skills 패키징 포맷

repo beta · 원문 · 3 sources

이번 주 GitHub Trending에서 같은 패키징 컨셉을 따르는 레포 세 개가 동시에 급상승했습니다. anthropics/skills가 138,380 스타로 한 주에 +4,749개, anthropics/financial-services가 26,286 스타로 +4,136개, K-Dense-AI/scientific-agent-skills가 24,973 스타로 +3,994개를 추가했습니다. 세 레포 모두 "Agent Skills" 라는 같은 단어를 쓰고, 같은 추상화 — 에이전트 런타임이 마운트해서 디스패치할 수 있는 사전 패키지된 작업 단위 — 를 발행합니다.

Skill의 구성

레포들을 열어보면 하나의 Skill은 대체로 세 가지로 이루어집니다.

  1. 선언 매니페스트 (YAML/JSON) — 이름, 설명, 기대 입력, 부수효과(파일 시스템·네트워크·외부 API 접근 여부), 필요 권한이 명시됩니다.
  2. 프롬프트 템플릿 — 에이전트가 이 Skill을 호출할 때 모델에게 주입되는 시스템/유저 메시지 조각입니다.
  3. 실행 코드 또는 워크플로 — Python 함수, 셸 스크립트, 또는 또 다른 LLM 호출 체인이 들어갑니다.

매니페스트 + 프롬프트 + 코드를 한 디렉터리로 묶어 배포한다는 점이 핵심입니다. 프롬프트 템플릿을 1급 객체로 다룬다는 부분이 LangChain의 Tool이나 OpenAI function-calling 스키마와 다른 지점입니다.

가장 가까운 기존 프로젝트와의 차이

가장 가까운 사전 프로젝트는 LangChain의 langchain.tools 모듈과 모델 컨텍스트 프로토콜(MCP) 서버입니다. LangChain은 실행 코드 중심이고 프롬프트는 호출 측에서 작성합니다. MCP는 프로토콜이라 패키징 단위가 "서버"이고, 한 서버가 여러 툴을 노출합니다. Agent Skills는 그 사이를 노립니다 — MCP보다 가벼운 패키징, LangChain보다 명시적인 프롬프트 분리입니다.

도입 전에 확인해 둘 부분

세 레포의 매니페스트 스키마는 세부에서 갈립니다. 권한 모델이 다르고, 비동기 실행 선언 방식이 다릅니다. 실행 모델도 다릅니다 — K-Dense-AI는 인-프로세스 코드로 실행하는 예시가 많고, anthropics/skills는 외부 프로세스/샌드박스 가정이 강합니다. 주간 +13k 스타 규모의 성장은 호환성 깨짐이 잦다는 뜻이기도 하므로, 도입한다면 핀 + CHANGELOG 모니터링이 필수입니다.

이번 주가 의미하는 것

핵심 신호는 어느 한 레포의 스타 수가 아닙니다. 서로 독립된 세 조직(모델 랩 하나, 그 랩의 도메인 사이블링 하나, 외부 커뮤니티 그룹 하나)이 같은 패키징 모양에 수렴했다는 점입니다. 플러그인 시스템을 각자 발명하던 단계는 지나가고, 포맷 자체가 조율 지점이 되기 시작했다고 볼 수 있습니다.

도입을 고려한다면 anthropics/skills의 매니페스트 두세 개만 먼저 읽어보세요.

"지루한 프로덕션" 단계로 진입한 코딩 에이전트

blog_post production-ready · 원문 · 3 sources

이번 주에 서로 다른 출처의 세 글이 같은 결론에 도달했습니다. 코딩 에이전트가 더 이상 데모 단계가 아니라는 결론입니다.

Simon Willison의 PyCon 라이트닝 토크는 변곡점을 2025년 11월로 명시합니다. TheNewStack의 Conductor Cloud 보도는 코딩 에이전트가 호스팅된 환경에서 수 시간 동안 비감독으로, 여러 인스턴스가 병렬로 도는 패턴을 묘사합니다. OpenAI Codex의 ChatGPT 모바일 통합은 데스크톱과 원격 SSH를 휴대폰에서 잇고, 자격증명은 별도의 보안 릴레이를 통해 전달됩니다.

세 글이 따라가는 사실들

Willison은 2025년 11월을 변곡점으로 본 근거를 두 가지로 제시합니다. 첫째, 같은 모델군이 6개월 동안 같은 코드베이스에서 일관된 PR 머지율을 유지했습니다. 둘째, 로컬 모델이 같은 벤치마크에서 클라우드 모델과의 격차를 좁혔습니다. 두 근거 모두 정량적 수치를 제시하지는 않지만, 추세선의 방향은 명확합니다.

Conductor Cloud는 다른 각도에서 같은 결론에 도달합니다. 데스크톱 에이전트는 사용자가 노트북을 닫으면 죽는다. 호스팅 환경에서는 죽지 않는다. Conductor는 이 단순한 차이가 "몇 분짜리 작업"과 "몇 시간짜리 작업" 사이의 분기점이라고 봅니다. 보도에는 동시 실행 인스턴스 수나 평균 작업 시간 같은 구체적 수치는 빠져 있습니다.

Codex 모바일 발표는 세 번째 각도입니다. 자격증명을 휴대폰에 두지 않고도 데스크톱·SSH 환경에 에이전트를 풀어둘 수 있도록 보안 릴레이를 끼웠습니다. PR 리뷰나 빌드 트리거를 출퇴근길에 처리하는 시나리오를 향한 설계입니다.

세 글이 공통으로 가리키는 운영 패턴

세 글 모두 동일한 운영 패턴에 수렴합니다 — 장시간 실행, 자격증명 격리, 비동기 감독. 2024년식 "IDE에 붙어있는 인라인 어시스턴트"와는 다른 모드입니다. 어느 글도 이번 주에 에이전트가 더 똑똑해졌다고 주장하지 않습니다. 주장하는 건 주변 배관이 프로덕션 등급이 됐다는 점입니다. Willison만 단독으로 강한 주장을 펼치고, 두 제품 글은 출시 증거로 그 주장을 뒷받침합니다.

작년 동시점과의 차이

2024년 GitHub Copilot Workspace나 초기 Devin 데모 시절의 프레이밍은 "AI가 개발자를 대체한다"였습니다. 이번 주의 글들은 톤이 다릅니다. 에이전트는 비동기 워커, 사람은 supervisor라는 관계를 전제로 합니다. 모델 능력 자체에 대한 강한 클레임은 오히려 줄었습니다.

하나 짚고 갈 부분

TheNewStack의 두 글은 출시 발표 시점의 클레임을 그대로 옮기고 있고, 독립 벤치마크는 없습니다. Conductor의 "더 긴 자율 작업"에는 공개된 신뢰성 지표가 없고, Codex 모바일 릴레이는 3자 감사 결과가 없습니다.

자기 인프라에 적용한다면 "자격증명 브로커가 1시간짜리 비감독 작업을 가정하고 설계됐는지" 한 가지부터 확인해보세요.

평범한 인프라처럼 보이기 시작한 에이전트 인프라

paper experimental · 원문 · 2 sources

이번 주에 서로 매우 다른 두 출처가 같은 — 그리고 그동안 방치돼 온 — 레이어를 동시에 두드렸습니다. 에이전트를 굴리는 비용과 에이전트가 실행될 때 무엇을 건드리는지의 권한 모델입니다.

논문 "Temporal Semantic Caching and Workflow Optimization in Agentic Plan-Execute Pipelines"는 플래너·실행기 중간 출력을 시맨틱 유사도와 TTL로 캐싱해, 거의 동일한 서브문제를 다시 만나는 LLM 호출을 중복 제거하자고 제안합니다. 산업 쪽에서는 NanoCo의 NanoClaw가 직원당 하나씩 Docker로 격리된 에이전트를 배치하고, 자격증명을 직원 신원에 바인딩합니다.

논문이 실제로 제안하는 메커니즘

Temporal Semantic Caching의 작동 방식은 두 단계입니다. 첫째, 플랜-실행 파이프라인의 각 노드 출력(plan, sub-plan, tool call result)을 임베딩 + TTL 쌍으로 인덱싱합니다. 둘째, 같은 노드 위치에 새 입력이 들어왔을 때, 유사도가 임계값을 넘고 TTL이 만료되지 않은 캐시 엔트리가 있으면 재사용합니다. 단순히 "같은 쿼리면 재사용"이 아니라 "의미적으로 같은 서브문제"를 잡는다는 점이 핵심입니다.

주장하는 절감폭은 파이프라인당 비용 30~50%입니다. 다만 비교 기준이 무캐시 베이스라인이라는 점은 짚고 가야 합니다. 누구도 프로덕션에서 무캐시로 굴리지는 않으므로, 헤드라인 수치는 상한값으로 받아들이는 게 안전합니다.

방법론적으로 흥미로운 부분은 TTL을 워크로드별로 학습한다는 점입니다. 빠르게 변하는 외부 도구(예: 검색 API) 결과는 TTL이 짧고, 결정론적 계산 결과는 깁니다. 이걸 손으로 정하는 대신 "같은 노드에서 캐시 미스가 났을 때의 답이 얼마나 달랐는지"로 추론합니다.

NanoCo가 실제로 배포한 것

NanoClaw는 텔아비브 스타트업이 발표한 매니지드 에이전트 플랫폼입니다. 운영 모델이 평범한 SaaS 에이전트와 다릅니다.

  • 직원 단위 컨테이너: 에이전트 1대 = 직원 1명. Docker로 격리.
  • 자격증명은 단명: 컨테이너에 영구 박혀있지 않고, 매 작업마다 짧은 토큰을 발급받습니다.
  • 권한 확장에는 사람 승인이 필요: 직원 A의 에이전트가 직원 B의 메일에 접근하려면 명시적 위임이 필요합니다.

기존 멀티 테넌트 SaaS 에이전트는 보통 "에이전트 1대 = 워크스페이스 1개" 구조이고, 자격증명이 워크스페이스 단위로 한 통에 모입니다. NanoCo의 답은 그 통을 직원 단위로 쪼개는 모델입니다.

두 항목이 같은 클러스터에 묶이는 이유

이번 주는 연구와 제품이 드물게 잘 정렬돼 있습니다. 논문은 비용 축을, NanoCo는 접근 제어 축을 공격합니다. 두 쪽 모두 에이전트를 "모델만 좋으면 끝"인 객체가 아니라, 런타임 특성(캐시 히트율, 자격증명 표면적)이 모델만큼 중요한 장기 실행 객체로 다룬다는 공통점이 있습니다.

작년 동시점과의 차이

작년 같은 시점에 "에이전트 인프라"는 대체로 LangChain·LlamaIndex 같은 프레임워크나 벡터 DB를 가리키는 단어였습니다. 캐싱은 LLM 호출 레벨의 단순한 "prompt → response" 키-값 캐시였고, 권한 모델은 거의 논의가 없었습니다. 이번 주의 두 항목은 그 레이어 위로 한 칸 올라가 워크로드 단위 캐싱과 신원 단위 격리를 다룹니다.

아직 풀리지 않은 부분

논문은 적대적 입력 하에서의 캐시 무효화를 다루지 않습니다. 의도적으로 임베딩 거리에서 가깝지만 의미가 다른 입력을 던지면 잘못된 캐시 히트가 발생할 수 있습니다. NanoCo 보도는 두 직원의 에이전트가 상태를 공유해야 하는 케이스(예: 공유 프로젝트)를 어떻게 처리하는지 다루지 않았습니다.

자기 시스템에서는 "플래너 출력을 LLM 호출 캐시 위에 한 겹 더 캐싱했을 때 비용이 얼마나 줄어드는지" 한 가지부터 측정해보세요.
