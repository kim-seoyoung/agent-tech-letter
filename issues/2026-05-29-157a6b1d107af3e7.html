Claude Opus 4.8 출시와 Anthropic의 한 주

blog_post production-ready · 원문 · 2 sources

Anthropic이 5월 28일에 Claude Opus 4.8을 공개했어요. Simon Willison은 같은 날 올린 리뷰(https://simonwillison.net/2026/May/28/claude-opus-4-8/)에서 이 모델을 "a modest but tangible improvement"이라고 정리했어요. 헤드라인 숫자는 SWE-bench Verified 88.6%로, 4.7의 87.6% 대비 1.0%포인트 올라간 수준이에요. 컨텍스트 윈도우는 100만 토큰, 출력 한계는 128,000 토큰으로 이전과 동일하고, 지식 컷오프와 학습 데이터 컷오프 모두 2026년 1월로 그대로예요. 가격도 입력 $5 / 출력 $25 per million tokens로 4.7과 같은 라인을 유지했어요.

기능 쪽에서 의미 있는 변화는 두 가지예요. 첫째, messages 배열에서 user 턴 직후에 system 메시지를 허용해서, 긴 대화 도중 전체 시스템 프롬프트를 다시 보내지 않고도 지시를 갱신할 수 있게 됐어요. 캐시 친화적인 구조라 긴 에이전트 루프에서 토큰 소모를 줄일 여지가 생겨요. 둘째, TechCrunch가 보도한 "dynamic workflow" 도구가 추가됐는데, 모델이 자기 작업을 단계로 쪼개서 순차 실행하도록 만든 변경이에요.

신뢰성 지표도 따라 짚어볼 만해요. Anthropic은 "근거 없는 결과를 그대로 보고하는 비율"이 0%였고, 과신 비율도 Opus 4.7 대비 큰 폭으로 줄었다고 보고했어요(원문 표현으로는 "more than ten-fold reduction in overconfidence"). 이전 세대에서는 SWE-bench를 핵심 신호로 봤다면 이번엔 reliability 축이 함께 강조됐다는 점이 가장 큰 차이예요. Simon Willison은 외부 리더보드에서도 4.8이 4.7을 근소하게 이긴다는 점을 언급하면서도 "modest"라는 단어를 굳이 붙여 한 세대 안에서의 미세 조정에 가깝다고 평가했어요. 같은 주에 Anthropic 연환산 매출 런레이트가 470억 달러까지 올라왔다는 보도(https://simonwillison.net/2026/May/29/anthropic/)도 나왔으니, 모델 자체의 진폭과 사업적 모멘텀의 진폭을 분리해서 읽을 필요가 있어요. 직접 확인해볼 만한 한 가지는 user 턴 뒤의 system 메시지를 활용해 긴 에이전트 세션의 시스템 프롬프트 갱신을 캐시 친화적으로 바꿔보는 거예요.

Daytona의 베어메탈 에이전트 샌드박스

blog_post production-ready · 원문 · 2 sources

용어부터 짧게 짚으면, 여기서 sandbox는 에이전트가 코드 실행·셸 명령·파일 변경처럼 외부에 영향을 미치는 동작을 호스트와 분리된 임시 컴퓨트 환경 안에서만 돌리도록 떼어낸 격리 박스예요. LLM이 자율적으로 도구를 호출하기 시작하면 한 번의 실패가 시스템을 망가뜨릴 수 있기 때문에, "실패해도 호스트엔 흔적이 안 남는" 일회용 컴퓨터를 매번 새로 띄우는 게 에이전트 인프라의 기본 단위로 굳어지고 있어요.

Latent Space의 5월 21일 공개 에피소드 "Giving Agents Computers"(https://www.latent.space/p/daytona)에서 Daytona CEO Ivan Burazin이 에이전트 컴퓨트 인프라의 현 상태를 자세히 풀었어요. 숫자부터 보면, Daytona는 월 74%의 MoM 성장률을 유지하고 있고, 단일 최대 고객 한 곳에서만 하루 850,000번의 샌드박스 런이 돌아가요. 동시 가동 CPU도 50만 코어 단위까지 요청이 들어온다고 해요. 샌드박스 한 개의 프로비저닝 시간은 60밀리초 미만으로, 일반적인 컨테이너 기반 런타임 대비 한 자릿수 더 빠른 영역이에요.

핵심 아키텍처 결정은 베어메탈 위에 자체 스케줄러를 얹은 거예요. EBS 같은 네트워크 스토리지를 거치지 않고 머신의 디스크·CPU·RAM을 그대로 쓰기 때문에 IOPS가 빠르고, 에이전트가 코드·셸 명령·컴퓨터 사용 워크플로를 실행할 때 OS·디스크·메모리까지 제어할 수 있어요. Burazin은 이 구조를 "그냥 sandbox"가 아니라 "composable computer"라고 부르는 게 더 정확하다고 표현했어요.

흐름을 더 큰 그림에 놓고 보면, Anthropic이 5월 19일 London "Code with Claude"에서 self-hosted sandbox와 MCP tunnel을 공개하면서 관리형 샌드박스 제공자로 Cloudflare, Daytona, Modal, Vercel을 함께 명시했어요(https://thenewstack.io/anthropic-mcp-tunnels-sandboxes/). 즉 "오케스트레이션은 Anthropic, 도구 실행은 별도 인프라"로 갈라놓는 패턴이 표준 후보로 굳어지는 중이에요. 가장 가까운 대안은 e2b, Modal, Cloudflare Workers 샌드박스로, 베어메탈 가속 대신 컨테이너·VM 기반 격리를 우선하는 접근이에요. Daytona가 RL eval과 학습 인프라 영역까지 사용처를 넓히고 있다는 점에서 에이전트 컴퓨트와 RL 인프라가 합쳐지는 신호로도 읽혀요. 자체 인프라에 적용해본다면 평균 sandbox lifetime이 1분 미만인지부터 한 번 측정해 베어메탈/컨테이너 선택의 출발점으로 삼아보세요.

Datasette Agent 알파 공개

blog_post experimental · 원문 · 2 sources

Datasette 자체부터 짧게 짚으면, 같은 Simon Willison이 만든 오픈소스 도구로 SQLite 파일을 그대로 웹 위에 띄워 탐색·필터·JSON API·공유까지 한 번에 제공하는 데이터 퍼블리싱 플랫폼이에요. 권한 모델과 플러그인 생태계가 이미 갖춰져 있어서, 이번 알파는 그 위에 LLM 에이전트 레이어를 한 장 더 얹은 형태로 자연스럽게 확장된 셈이에요.

Simon Willison이 5월 21일에 Datasette Agent 첫 알파를 공개했어요(https://simonwillison.net/2026/May/21/datasette-agent/). Datasette 위에 얹는 오픈소스 플러그인이고, SQLite 데이터베이스에 대해 자연어로 질문을 던지면 LLM이 SQL 도구를 호출하면서 답하는 구조예요. 라이브 데모는 agent.datasette.io에서 돌아가고 있고, 기본 모델로 Gemini 3.1 Flash-Lite가 들어가 있어요.

구현 측면에서 의미 있는 결정은 LLM 추상화 레이어로 Simon이 그동안 만들어온 llm 패키지를 그대로 재사용한 거예요. 덕분에 OpenAI·Anthropic·Google Gemini 같은 프런티어 모델뿐 아니라 자체 하드웨어에서 돌리는 오픈 웨이트 모델까지, tool-calling을 지원하는 모델이라면 거의 다 백엔드로 끼울 수 있어요. Datasette 플러그인 시스템과도 잘 맞물려서, 시각화나 커스텀 도구를 추가하는 게 "플러그인을 하나 더 등록"하는 수준에 머물러요.

이 발표는 Datasette 1.0 알파 단계와 같이 묶여서 fixture 플러그인 흐름과도 짝을 이뤘어요. 시장에서 가장 가까운 비교군은 LangChain·LlamaIndex 위에서 만들어진 SQL 에이전트들과 Vanna AI 같은 NL2SQL 전용 도구예요. 차이는 두 가지인데, 하나는 Datasette Agent가 데이터 카탈로그·permission·플러그인 생태계를 이미 갖춘 Datasette 위에서 동작한다는 점이고, 다른 하나는 LLM 백엔드를 완전히 교체 가능한 추상화로 뽑아둔 점이에요. 결과적으로 "어떤 모델을 쓰느냐"는 운영 결정이 되고, 에이전트 코드 자체는 모델에 묶이지 않아요. 도입을 고려한다면 자기 Datasette 인스턴스에 plugin 하나만 설치해 라이브 데모와 같은 흐름으로 한 번 굴려보세요.
