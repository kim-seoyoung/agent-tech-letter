GLM-5.2 공개: 오픈 웨이트가 처음 GPT-5.5와 정면으로 맞붙은 한 주

blog_post production-ready · 원문 · 3 sources

Z.ai가 GLM-5.2를 6월 13일 자사 코딩 구독자에게 먼저 풀고, 6월 16일 전체 가중치를 MIT 라이선스로 Hugging Face에 올렸어요(https://simonwillison.net/2026/Jun/17/glm-52/). 총 파라미터 753B, 활성 40B의 MoE 구조예요. 컨텍스트 길이는 100만 토큰으로, 직전 GLM-5.1의 20만 토큰에서 다섯 배 늘었어요. Z.ai가 따로 강조한 항목은 IndexShare라는 어텐션 구조 최적화예요. 1M 컨텍스트 끝단에서 토큰당 연산량이 2.9배 줄었다고 보고했어요. 긴 트레이스를 그대로 들고 가야 하는 장기 실행 에이전트가 가장 먼저 체감할 마진이에요.

벤치마크 숫자가 이 모델의 위치를 또렷하게 잡아줘요. SWE-Bench Pro에서 62.1점을 받았어요. 같은 벤치마크에서 GPT-5.5가 58.6점이라, 오픈 웨이트가 폐쇄 프런티어를 메이저 코딩 평가에서 처음 추월한 사례로 기록됐어요. Terminal-Bench 2.1에서는 81.0점으로 Claude Opus 4.8의 85.0점과 4점 안쪽으로 좁혔어요. 장기 호흡 코딩 벤치마크 FrontierSWE에서는 74.4%로 GPT-5.5의 72.6%를 앞서고 Opus 4.8의 75.1%와 사실상 동률이에요. 도구 호출 벤치마크 MCP-Atlas에서도 77.0으로 GPT-5.5의 75.3과 Opus 4.8의 77.8 사이에 자리잡았어요. Artificial Analysis Intelligence Index에서는 오픈 웨이트 1위로 올라섰어요. VentureBeat는 동일 토큰당 비용이 GPT-5.5의 약 1/6 구간이라고 전했어요.

배포 경로는 두 갈래예요. 오픈 가중치는 Hugging Face에서 바로 내려받아 자체 인프라에 띄울 수 있어요. 호스팅 추론은 Z.ai 자신의 코딩 구독 외에 DeepInfra·OpenRouter·NVIDIA NIM에서 같은 주에 인덱싱됐어요. 다만 풀 파라미터 753B는 단일 H100 80GB 한 장에 다 올라가지 않아요. 한 분석은 "양자화하지 않은 가중치로 돌리려면 8장 묶음이 기본"이라고 적었어요. Latent Space는 6월 17~18일 AI News에서 여러 실사용자가 "프런티어에 인접한 첫 오픈 웨이트"라는 평을 모았다고 정리했어요(https://www.latent.space/p/ainews-glm-gpt-glm-52-passes-vibe).

선행 흐름과 견주면, 직전까지 같은 자리는 GLM-4.6·DeepSeek-V3.5·Meta Llama 4 405B가 차지하고 있었어요. 셋 다 같은 시점의 폐쇄 프런티어와 한 자릿수 후반대 격차가 있었어요. GLM-5.2는 그 격차를 코딩·도구 호출 같은 에이전트 워크로드에서 처음으로 한 자릿수 초반까지 좁혔다는 점이 핵심이에요. 6월 13일 미국 정부가 Claude Fable 5의 외국인 접근을 차단한 시점에서 일주일 차이라는 맥락도 같이 읽힐 만해요. 자기 스택에 적용해본다면, 운영 중인 코딩 에이전트의 24시간 트레이스 한 묶음을 GLM-5.2 호스팅 엔드포인트로 한 번 리플레이해 같은 도구 호출 순서가 재현되는지부터 한 줄로 비교해보세요.

Vercel eve: "Next.js for agents"라는 표지의 오픈소스 에이전트 프레임워크

repo beta · 원문 · 2 sources

Vercel이 6월 17일 Vercel Ship London에서 오픈소스 에이전트 프레임워크 eve를 공개했어요(https://github.com/vercel/eve). 회사 자신의 표현은 "Next.js for agents"예요. 한 에이전트가 한 디렉터리라는 점이 핵심 설계예요. 디렉터리 안에는 모델을 지정하는 파일, Markdown으로 적힌 시스템 프롬프트 파일, 도구 정의 파일, 평가용 파일이 약속된 이름으로 들어가요. Next.js가 페이지 파일을 디렉터리로 매핑한 패턴을 그대로 가져온 구조예요. AI Gateway·sandboxing·approval gate·채널 통합 같은 프로덕션 측 부품을 프레임워크 자체가 떠안아요.

번들이 어디까지 깔려 있는지가 적용 판단의 첫 갈래예요. 모델 호출은 Vercel의 AI Gateway가 받아 프로바이더 fallback을 처리해요. 코드 실행은 격리 샌드박스에서 일어나요. 일부 도구는 사람 승인이 필요하면 approval gate에서 멈춰 채널로 알림이 가요. 채널은 Slack·Linear·이메일 등으로 약속된 이름의 파일 하나만 추가하면 묶여요. 런타임은 durable execution을 기본값으로 가져가, 도구 호출 중간에 프로세스가 죽어도 같은 상태에서 재개할 수 있어요. 회사는 "운영 중인 프로덕션 에이전트 fleet 위에서 같은 프레임워크를 그대로 쓰고 있다"고 적었어요.

선택할 다른 프레임워크와 비교해 차이를 짚어둘 만해요. LangChain의 LangGraph는 같은 durable execution 모델을 가져가지만 Python 중심이고 서버리스에서는 굴리기 어려워요. AutoGen·CrewAI 계열은 멀티 에이전트 토폴로지의 표현력은 높지만 프로덕션 부품(샌드박스·approval·채널)을 직접 붙여야 해요. eve는 표현력에서는 한 단계 보수적이지만, 같은 디렉터리만 들고 가면 Vercel·Cloudflare Workers 같은 서버리스 위에서 그대로 돈다는 점을 트레이드로 들고 나왔어요. The New Stack은 "Vercel 스택 위에서 빠르게 프로덕션 에이전트를 굴리고 싶을 때의 기본값이지만, 포터빌리티가 단단한 요구사항이면 잘못된 기본값"이라는 톤으로 정리했어요(https://thenewstack.io/vercel-launches-eve-an-open-source-framework-that-treats-agents-as-directories/).

오픈소스가 갓 풀린 시점이라 채택 신호는 두 가지로 봐야 해요. 하나는 Vercel 자신의 dogfooding 폭이고, 다른 하나는 첫 외부 사례가 얼마나 빨리 붙는지예요. 한 디렉터리=한 에이전트라는 약속이 깔린 만큼, 같은 자리에서 "이미 굴리는 에이전트의 시스템 프롬프트와 도구 명세를 eve 디렉터리 한 개로 옮길 수 있는지" 자체가 가장 빠른 검증 동작이에요. 자기 스택에 적용해본다면, 운영 중인 에이전트 한 개를 골라 vercel/eve의 디렉터리 규약 그대로 옮긴 다음 같은 trace로 도구 호출 순서가 동일하게 재현되는지부터 한 번 돌려보세요.

Databricks LTAP: 에이전트가 읽고 쓰는 데이터의 그릇을 한 계층으로 합치기

blog_post production-ready · 원문 · 1 source

Databricks가 6월 16일 Data + AI Summit에서 LTAP(Lake Transactional/Analytical Processing) 아키텍처를 공개했어요(https://thenewstack.io/databricks-is-rebuilding-the-data-stack-for-ai-agents/). 한 줄로 줄이면 OLTP와 OLAP를 한 저장 계층에 합치는 설계예요. 작년 출시된 Lakebase(오픈 오브젝트 스토리지 위에서 도는 서버리스 Postgres)와 Lakehouse를 한 거버넌스 모델로 묶었어요. 그 결과 운영 데이터를 따로 ETL 없이 분석 워크로드에서도 같은 사본으로 읽을 수 있어요. 회사는 "수십 년 데이터 인프라를 정의해온 파이프라인·복제본·ETL 오버헤드를 빼는 게 목표"라고 적었어요.

숫자로 본 운영 규모는 다음과 같아요. Lakebase는 작년 출시 후 1년 만에 Block·Ensemble·Superhuman·Zillow를 포함한 수천 고객을 받았어요. 하루 1,200만 회의 데이터베이스 인스턴스 기동을 처리한다고 적었어요. 이번 발표에는 cross-cloud·cross-region 재해 복구가 새 기능으로 들어왔어요. 같은 트랜잭션 로그를 다른 클라우드·다른 리전에서 그대로 받아 페일오버를 돌릴 수 있는 구조예요. 트랜잭션 워크로드와 분석 워크로드는 같은 저장 계층 위에서 별도 컴퓨트로 떠 있어요. 한쪽 부하가 다른 쪽 지연을 끌어올리지 않도록 강한 격리를 유지한다고 적었어요.

에이전트와 연결되는 지점은 두 군데예요. 첫째, 자율 에이전트가 호출하는 쿼리는 보통 운영 DB의 최신 행과 분석 워크하우스의 집계 뷰를 같이 봐야 해요. 지금까지 둘 사이에는 분 단위 이상의 ETL 지연이 박혀 있었어요. LTAP는 같은 저장 사본을 두 쪽이 보게 해 그 지연을 0에 가깝게 누른다는 주장이에요. 둘째, Unity Catalog로 통합된 거버넌스가 RBAC·라인리지·감사 로그를 한 군데로 모아요. 에이전트가 어디까지 어떤 데이터를 봤는지 추적이 단순해져요. 지난주 다룬 Stack Overflow for Agents가 "에이전트끼리의 지식 풀"을 깔았다면, LTAP는 그 아래 "한 조직 안의 운영·분석 데이터 풀"을 같은 계층으로 모으는 시도예요.

가장 가까운 비교군은 Snowflake의 Unistore와 Microsoft의 Fabric IQ + HorizonDB 묶음이에요. Snowflake는 Hybrid Tables로 같은 방향을 시도하고 있지만 트랜잭션 워크로드 규모가 아직 제한적이에요. Microsoft 쪽은 같은 그림을 데이터 패브릭으로 풀지만 운영 DB는 별 엔진으로 두고 있어요. LTAP는 운영 DB(Postgres 호환 Lakebase)와 분석 엔진(Photon) 둘 다 자기 스택 위에 두는 점에서 격차를 노렸다고 읽혀요. 자기 조직에 적용해본다면, 운영 중인 에이전트가 같은 거래 한 건을 OLTP에서 읽은 직후 분석 뷰에서도 즉시 읽히는지 — 그 사이 지연이 몇 초인지부터 한 번 측정해보세요.
